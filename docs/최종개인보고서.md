## 2. 담당 작업 내용 설명

### 2-1. 음식 분류 파이프라인

**모델 구조**
EfficientNet V2-M을 백본으로 한 70-class 한국 식재료 분류 모델. Dropout(p=0.3)을 포함한 분류 헤드를 붙이고 AdamW 옵티마이저, ReduceLROnPlateau 스케줄러로 학습했다.

**Progressive Unfreezing**
단계적 해동 전략을 직접 설계해 적용했다. Epoch 5까지는 backbone 상위 4개 블록(`features[-4:]`)과 head만 학습(LR=1e-4), Epoch 5부터 backbone 전체 해동 후 새로 풀린 하위 레이어는 별도 param_group으로 LR=1e-5를 적용해 catastrophic forgetting을 방지했다.

**모델 진화**

| 버전 | 클래스 | val_acc | 주요 변경 |
|---|---|---|---|
| ver2 (5/3) | 35 (영문) | 94.56% | 초기 베이스라인 |
| ver3 (5/13) | 60 (영문) | 94.73% | 클래스 확장 |
| ver4 (5/27) | 70 (한글) | 95.42% | Progressive Unfreezing 도입 |
| ver5 (5/28) | 70 (한글) | 96.60% | patience 5→8, epoch 30→50 완화 |

ver4 학습 로그(`docs/logs/classifier/ver4_0527/training_log.csv`)를 분석하니 backbone 전체 해동이 epoch 6에 일어났고 epoch 7에 바로 최고점(95.42%) 달성 → patience=5 소진으로 epoch 12 조기 종료. 저장된 best 모델 기준 backbone은 단 2에폭만 학습된 상태였다. 이를 근거로 patience 5→8, epoch 30→50으로 완화해 ver5 재학습 → val_acc 96.60% (+1.18%p).

**Self-improving 구조**
소프트맥스 최댓값 80% 미만 시 Gemini 2.5 Flash Vision으로 자동 위임. Gemini는 전체 클래스 목록을 프롬프트로 받아 목록 내 클래스면 해당 클래스명, 없으면 자유 한국어 라벨 반환. 폴백된 이미지는 `dataset/auto_labeled/{label}/`에 자동 저장되어 다음 학습 라운드의 데이터로 자가 누적된다. 자문의 "VLM 쓰는 건 어때?"라는 제안을 ① 분류 → ② 저신뢰 VLM 위임 → ③ 라벨 자동 누적 → ④ 다음 학습 반영의 4단계 self-improving 구조로 직접 구체화했다.

**데이터 파이프라인**
공개 데이터셋에 한국 식재료가 없어 5단계 자동 파이프라인을 구축했다.

1. `data_crawl.py` — DuckDuckGo 크롤링
2. `data_clean.py` — MD5 해시 중복 제거
3. `data_split.py` — train/val 분리 (멱등성 보장)
4. `data_diet.py` — 클래스별 상한(train ≤ 400, val ≤ 80)
5. `data_len.py` — 분포 통계 출력

최종 데이터셋: train 12,714 / val 3,145장 (총 약 15,000장).
학습 시 `WeightedRandomSampler`로 클래스 불균형 보정, Label Smoothing(0.1) 적용.

### 2-2. 영수증 OCR 파이프라인

**2단계 구조**
- 1단계: Google Document AI로 영수증 텍스트 추출 및 한글 필터링
- 2단계: 추출된 텍스트를 Gemini 2.5 Flash에 전달, `response_mime_type="application/json"`으로 JSON 강제 응답. `purchasedAt`(YYYY-MM-DD)과 식재료만 필터된 `ingredients[]` 반환

**카테고리 정규화**
Gemini가 대소문자 혼용·오타·미정의 카테고리를 반환하는 경우를 방어하기 위해 `models/category.py`의 `normalize_category()`로 일원화. 허용 집합(`VALID_CATEGORIES`) 외 값은 모두 `ETC`로 보정.

**평가 결과**
- 구매일 추출: 4/4 (100%) — 날짜 없는 영수증도 `null` 정확 반환
- 품목: 대부분 정상 추출. 주요 오류 패턴은 상품명 그대로 출력(`마늘빅프랑크`, `초마짬뽕` 등) — 의미적으론 정확하나 표기 정규화 후처리 필요

### 2-3. 냉장고 감지 파이프라인

**구조**
Gemini 2.5 Flash Vision에 냉장고 사진 1장을 전달해 식재료 목록 JSON을 반환받는 단일 호출 구조. YOLO 기반 구조 검토 후 ① COCO 한국 식재료 미포함 ② 다중 호출 지연 ③ YOLO-World 한국어 불안정 세 가지 이유로 Gemini Vision 단독으로 전환했다.

**프롬프트 개선**
초기에 "즉석식품", "음식" 같은 포괄어가 출력되는 문제를 발견. 블랙리스트 방식에서 원칙 기반("구체 품목명이 아닌 카테고리성 단어는 절대 쓰지 마, 확신할 수 없으면 빼라") 방식으로 전환해 오검출을 줄였다.

**평가 결과**
- 잘 보이는 병류·과일류: 안정적 인식
- 누락 패턴: 서랍·계란 가드·불투명 용기 안 식재료(물리적으로 안 보이는 문제)
- 오검출 패턴: 빽빽한 사진에서 없는 항목 출력(감자튀김, 컵라면 등)

---

### 2-4. FastAPI 서버 통합

3종 파이프라인을 단일 FastAPI 서버에 통합했다. 주요 설계 결정:

- **동기 엔드포인트**: `async def` → `def` 전환으로 torch 추론·Gemini 호출 같은 블로킹 작업이 이벤트 루프를 점유하는 문제 해결. FastAPI가 외부 스레드풀에서 처리해 요청 병렬 처리 가능
- **MPS 멀티스레드 방지**: 음식 분류 추론에 `threading.Lock` 적용. OCR·냉장고는 네트워크 I/O라 잠금 없이 병렬 유지
- **보안**: Bearer 토큰 + `hmac.compare_digest()` 타이밍 공격 방어, 실제 파일 magic bytes 이중 검증으로 Content-Type 헤더 위조 방어
- **타임아웃**: SDK 네이티브 `HttpOptions(timeout=...)` 로 Gemini 무한 대기 방지
- **하드웨어 자동 감지**: MPS → CUDA → CPU 순 자동 선택. AMP는 CUDA에서만 활성화

## 3. 주요 이슈 및 해결 내용

### 이슈 1 — 크롤링 데이터와 실사 사진의 분포 차이

**문제**: 학습 데이터는 정면·고화질 크롤링 이미지인데, 실제 앱 사용자는 기울어지거나 흐릿한 스마트폰 사진을 올린다.

**해결**: RandomPerspective(비스듬한 각도), GaussianBlur(초점 흐림), ColorJitter(조명 변화) 등 실제 촬영 환경을 시뮬레이션하는 증강을 추가해 별도 실사 데이터 없이 분포 차이를 줄였다.

---

### 이슈 2 — 신뢰도 임계값 trade-off

**문제**: 클래스가 70개로 늘어나면서 소프트맥스 최댓값이 자연적으로 낮아져 임계값 80% 기준에서 폴백이 잦아졌다. 임계값을 75%로 낮추니 신뢰도 77%짜리 오분류가 그대로 통과됐다.

**해결**: 임계값 80%로 복원. 임계값 조정이 아닌 모델 자체 개선(ver5)으로 방향을 전환해 폴백 빈도를 낮췄다.

---

### 이슈 3 — 정확한 평가 척도 부재 → 클래스별 지표 분석 및 개선 실험

**문제**: 초기에는 val_acc 단일 수치만으로 모델을 평가했다. 96.60%라는 수치만으로는 어떤 클래스에서 오류가 집중되는지, 오류의 방향이 무엇인지 파악할 수 없었다. 성능 개선을 시도하려 해도 무엇을 개선해야 하는지 특정이 불가능한 상태였다.

**해결**: `eval_val_V2_M.py`를 구현해 클래스별 Precision·Recall·F1과 혼동행렬을 산출하는 평가 체계를 추가했다. 분석 결과 Macro P 96.8% / R 96.6% / F1 96.6%이었으며, 오류가 무작위가 아니라 시각적으로 유사한 특정 쌍에 집중됨을 확인했다. 치즈는 Recall 74.4%로 최하위였고 오류의 40%가 버터로 분류됐으며, 돼지고기 오류 대부분은 소고기·닭고기에 몰렸고, 햄은 Precision 85.5%로 다른 클래스 오답이 햄으로 몰리는 FP 자석 현상이 있었다. 이를 기반으로 LR을 1e-4→5e-5로 낮춰 재학습을 시도했으나, 같은 클래스 쌍에서 동일한 혼동이 반복됐다. 하이퍼파라미터 조정으로는 해결되지 않으며 학습 이미지 자체의 시각적 유사성이 원인임을 규명했다.

**결론**: 단일 정확도 지표로는 문제를 특정할 수 없었고, 클래스별 P/R/F1 평가 체계를 갖추고 나서야 오류 원인을 데이터 수준까지 파악할 수 있었다. 개선 방향은 추가 튜닝이 아닌 혼동 쌍 타깃 데이터 보강으로 결론지었다.

---

### 이슈 4 — YOLO 냉장고 감지 한계

**문제**: YOLOv8n(COCO 80 클래스)에 한국 식재료 대부분이 미포함. YOLO + EfficientNet 연계안은 냉장고 내 다수 아이템에서 Gemini 다중 호출이 발생해 응답 지연이 심각했다. YOLO-World는 한국어 식재료 인식률이 불안정했다.

**해결**: Gemini 2.5 Flash Vision 단독 사용으로 전환. 사진 1장 → 1회 호출 → JSON 반환 구조로 단순화. 한국 식재료 인식률 가장 높고 학습 데이터 0장으로 운영 시작 가능.

---

### 이슈 5 — Gemini 응답 포맷 변경

**문제**: Gemini 2.5가 단일 객체를 `[{...}]` 배열로 감싸 반환하면서 운영 중 `'list' object has no attribute 'get'` 에러 발생.

**해결**: `isinstance(result, list)` 가드로 배열이면 첫 번째 요소 추출, dict가 아니면 에러 반환하는 방어 로직 추가.

---

### 이슈 6 — 냉장고 포괄어 오검출

**문제**: "즉석식품", "음료수", "채소" 같은 카테고리성 단어가 식재료로 출력됐다. 블랙리스트 방식은 목록에 없는 새 포괄어를 막지 못했다.

**해결**: 원칙 기반 프롬프트로 전환. "구체 품목명이 아닌 모든 카테고리성 단어는 절대 쓰지 말 것, 확신할 수 없으면 그 항목을 빼라"는 원칙으로 교체 후 포괄어 오검출이 눈에 띄게 감소했다.

## 5. 개인 기여 내용 및 배운 점

### 개인 기여 내용

본 프로젝트의 AI 파이프라인 전체를 단독으로 설계·구현했다. 구체적 기여는 다음과 같다.

**모델 설계 및 학습**
- EfficientNet V2-M 기반 70-class 한국 식재료 분류 모델 설계 및 4회 반복 학습(ver2~ver5)
- Progressive Unfreezing 파인튜닝 전략 직접 설계·적용
- 15,000장 규모 자체 데이터셋 크롤링·선별·분할 자동화 파이프라인 구축

**아키텍처 설계**
- 신뢰도 기반 Gemini 폴백 + Self-improving 4단계 구조 직접 설계
- 3종 파이프라인(분류·OCR·냉장고)을 단일 FastAPI 서버에 통합
- 동기 엔드포인트·스레드 락·SDK 네이티브 타임아웃 등 운영 안정성 설계

**평가 체계 구축**
- 분류 혼동행렬 기반 클래스별 P/R/F1 분석 도구 구축
- 영수증·냉장고 정답셋 평가 도구(`eval_common.py`·`eval_receipt.py`·`eval_fridge.py`) 직접 설계·구현
- 단일 F1 점수 대신 '예측 vs 정답' 비교표 방식 채택 — 표기 차이와 실제 오인식을 분리해 정확한 원인 진단

### 배운 점

**데이터 품질이 모델 성능을 좌우한다**
LR을 1e-4→5e-5로 조정해도 동일한 클래스 쌍에서 동일한 혼동이 반복됐다. 치즈↔버터, 돼지고기↔소고기처럼 학습 이미지 자체가 시각적으로 유사한 것이 원인이었고, 이는 추가 튜닝으로는 해결되지 않는다. 모델 성능의 실질적 병목은 모델 구조나 하이퍼파라미터가 아니라 데이터 품질임을 직접 확인했다.

**평가 체계 없이는 개선 방향을 잡을 수 없다**
val_acc 단일 수치만 보던 단계에서는 모델이 어느 클래스에서 왜 틀리는지 알 수 없었다. 클래스별 P/R/F1과 혼동행렬을 추가하고 나서야 "치즈 오류의 40%가 버터"라는 구체적인 원인을 특정할 수 있었고, LR 조정 실험으로 하이퍼파라미터가 원인이 아님을 확인하며 데이터 보강이 정공법임을 결론지을 수 있었다. 무엇을 측정하느냐가 무엇을 개선하느냐를 결정한다는 것을 배웠다. 또한 학습 로그를 버전별로 보존하는 것이 사후 원인 분석에 직접적으로 활용됐다.

**모델 하나보다 파이프라인 설계가 더 중요하다**
분류 정확도 96.6%보다, 그 모델을 어떻게 서버에 통합하고 신뢰도 기반 폴백을 어떻게 연결하며 운영 중 데이터를 어떻게 누적할 것인지 설계하는 과정이 더 많은 판단을 요구했다. AI 서비스에서 모델 성능은 필요조건이지 충분조건이 아니라는 것을 실감했다.

**외부 LLM은 방어 코드 없이는 운영할 수 없다**
Gemini는 응답 포맷을 예고 없이 바꾸고, 카테고리 값에 오타를 섞고, 포괄어를 식재료로 반환했다. JSON 강제(`response_mime_type`), `isinstance` 가드, 정규식 재검증, 카테고리 정규화 모듈, 타임아웃 설정 등 다중 안전장치를 실제 운영 중 문제를 겪으며 하나씩 추가했다. 외부 API는 신뢰하되 검증하는 방어적 설계가 필수임을 배웠다.

**평가 설계도 개발의 일부다**
단일 F1 점수로 영수증·냉장고를 평가하면 표기 차이(상품명 vs 일반명)가 오인식으로 잡혀 실제 문제를 가린다. '예측 vs 정답' 비교표 방식으로 바꾸고 나서야 "인식이 안 된 것"과 "표기만 다른 것"을 분리해 정확한 보완 방향을 도출할 수 있었다. 어떻게 평가할지를 잘못 설계하면 잘못된 결론에 이른다는 것을 경험했다.
